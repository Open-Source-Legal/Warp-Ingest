# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`warp-ingest` is a **deterministic, rule-based PDF parser** (Python package `warp_ingest`). It turns a PDF into layout-aware blocks (sections, paragraphs, lists, tables) with accurate bounding boxes and parent↔child relationships, rendered as blocks, JSON, HTML, or an OpenContracts structural export. It is **PDF-only** — non-PDF input is rejected (`ingestor_api.UnsupportedMediaType`). The design is deliberately rule-based rather than model-based (see README "Rule-based vs. model-based") — favor rules over ML when extending it.

## Pure-Python: no Java, no Tika

This codebase **used to** depend on a Java/Apache-Tika server (an nlm-modified jar) for PDF extraction and Tesseract for OCR. That is gone. The PDF front-end is now pure Python:

- **`pdfplumber`** (MIT, on pdfminer.six) extracts per-word bounding boxes + font data.
- **`rapidocr-onnxruntime`** (Apache-2.0; optional `[ocr]` extra) does OCR — the PP-OCR models on `onnxruntime`, weights bundled in the wheel. No Tesseract binary, no GPU.
- **`pypdfium2`** (Apache/BSD) renders pages to images for OCR.

The crucial design point: the front-end **reproduces the exact intermediate XHTML format the old Tika produced**, so the entire downstream rule engine (`visual_ingestor` and friends) was left untouched. When changing extraction, preserve that XHTML contract (see Architecture) rather than editing the layout engine.

## Commands

Uses `uv` and Python ≥3.10.

```bash
# install (core deps + dev tools, then with the optional OCR backend)
uv sync --group dev
uv sync --group dev --extra ocr
uv run python -m nltk.downloader punkt punkt_tab stopwords   # one-time

# lint / format (CI runs `make check`)
make format          # black . && isort .
make check           # black --check && isort --check  (alias: make lint)

# tests — pure Python, NO Java/Tika server needed
make test                                   # uv run pytest -vvs .
uv run pytest tests/                        # same, directly
uv run pytest tests/test_fixtures_parse.py -q      # parse every fixture (incl. OCR) end-to-end
uv run pytest tests/test_line_parser.py -k incomplete_line   # single test

# run the service (FastAPI/uvicorn, port 5001; concurrency autotuned from CPUs)
python -m warp_ingest.ingestion_daemon     # or ./run.sh

# docker (pure-python image)
make build && make run                      # Dockerfile.test, maps 5010->5001
```

## Architecture

### Request flow
`ingestion_daemon/app.py` (FastAPI) exposes `POST /api/parse`, API-key-gated (`WARP_API_KEY`, fallback `abc123`; send `X-API-Key` or `Authorization: Bearer`). Query params (booleans accept `true/false/1/0/yes/no/on/off`): `render_format` (`all`|`json`|`html`|`opencontracts`), `apply_ocr` (forces OCR on every page; scanned pages are OCR'd automatically regardless), `disable_ocr` (per-request: every page keeps its embedded text layer; conflicts with `apply_ocr` → 422), `semantic_units`. The response body is the parse result (`page_dim`/`num_pages`/`result`); errors are standard `{"detail": ...}`. It calls `ingestor/ingestor_api.py::ingest_document()`: `application/pdf`→`PDFIngestor`; any other MIME type raises `UnsupportedMediaType` (the daemon returns **415**). `GET /` and `GET /healthz` are unauthenticated health endpoints. The launcher (`ingestion_daemon/__main__.py`, via `python -m warp_ingest.ingestion_daemon` / `run.sh`) autotunes concurrency from the effective CPUs (affinity + cgroup quota, `ingestion_daemon/autotune.py`): one uvicorn worker per CPU by default, with `WARP_FE_WORKERS`/`WARP_OCR_THREADS`/`WARP_WORKER_PARSE_SLOTS` budgeted so web workers × FE pool × onnx threads never oversubscribe; every knob is env-overridable (`WARP_WEB_WORKERS`, `WARP_HOST`, `WARP_PORT`). Service tests: `tests/test_service.py` (TestClient, in-process). Warp-Ingest is PDF-only.

### OpenContracts export (`render_format=opencontracts`, PDF only)
`ingestor/opencontracts_exporter.py::to_opencontracts_export(xhtml, blocks)` converts the **already-produced** Tika XHTML + finalized `blocks` into an `OpenContractsDocExport` dict — it reads engine output only, never edits `visual_ingestor`. PAWLS word tokens are rebuilt from the XHTML `word-start/end-positions` (so **OCR text is included → no content loss**); one structural `TOKEN_LABEL` annotation per block; `annotation_json` is shape-A with `bounds` = union of the block's assigned tokens. `validate_export()` enforces the format's invariants. Helper `pdf_ingestor.parse_to_opencontracts(path)` runs the whole pipeline to an export. Exporter-level layout corrections (the sanctioned place — they only affect the export, not the `json`/`html`/`all` renders or the S-1 suite):
- **Label demotion** (`_resolve_label`): a header-ish block is relabeled `Paragraph` when its text is over-long (> `_HEADER_MAX_WORDS`=12 words), its *assigned token count* exceeds `_HEADER_MAX_TOKENS`=20, **or** its text is page furniture / processing metadata (`_is_nonheading_furniture`). The first two catch *run-in headings* ("6.6 Taxes. Each Party…") where a bold lead-in absorbed the section body; the token-count rule catches the case the word rule can't see (a 2-word lead-in whose box absorbed 188 body tokens). The furniture rule (from the 60-page audit) is **structural / format-only**: it demotes a header whose *entire text is a single non-prose structural token* — a page folio (`105`, `iii`, `F-18`, `- 33 -`), a bare email, or a bare URL — none of which is a heading, and which (worse) adopted body as children when mislabeled (page folios splitting a section across a page break were the audit's #1 clean win). Demotion fixes the label *and* the parenting cascade (tree consistency then splices the non-heading out); the same filter gates the re-segmentation path (`_is_centered_heading_line`). The rule deliberately uses **structure/format, not document content**: it does *not* match content phrases (recital openers, signature-field words, `TABLE OF CONTENTS`, e-signature watermarks, addresses) so the rule set never overfits to a contract's words. Justified by the Docling study + the 36- and 60-page heuristic audits; verified neutral against the Docling oracle (`Page N of M` folios and 4-digit years are deliberately *not* demoted — they perturb the head-ancestor oracle on cerebras_ex1013e — a baseline-regeneration deferral).
- **Geometric corner-furniture demotion** (`_resolve_label` via `_is_corner_furniture`, geometry passed from `to_opencontracts_export`): a `Section Header` whose assigned-token box sits in the page's **extreme top/bottom margin** (`top_frac < 0.055` or `bottom_frac > 0.91`) **and** is **horizontally offset past page center** (`left_frac > 0.55`) is a running header/footer / page-number / corner filing stamp — never a section heading (those start at the left text margin or are centered). This is the *structural* (position-only, content-free) counterpart that catches the corner stamps the text filter won't name (the FortWorth `CSC No.` / `OFFICIAL RECORD` clerk stamps), fixing their parenting cascades. `left_frac > 0.55` excludes every left-aligned/centered real heading by construction; the top band is kept tight (0.055) so the slightly-lower top-right `Exhibit X.X` identifier line is spared (demoting it is defensible but perturbs the small-doc Docling oracle — left to a baseline-regen follow-up). Verified **exactly Docling-neutral** (only the pre-existing `spacex_ex41` env-drift fixture fails). Validated by a 36-doc corner scan + per-fixture Docling A/B. **Fix 2 adds a deep-far-corner case**: a box with `bottom_frac > 0.88` **and** `left_frac > 0.70` is also demoted, catching the bottom-right clerk-stamp footer (fw_vertigis `OFFICIAL RECORD / CITY SECRETARY / FT. WORTH, TX`) that sits just *above* the 0.91 band; the tighter x-gate (0.70) keeps every left/centered heading safe.
- **Prose / recital pseudo-header demotion** (Fix 2 — `_is_prose_header` in `_resolve_label`; root-caused in the engine via `Line.is_prose_continuation`): a `Section Header` whose text is a *sentence or clause* rather than a label is demoted to `Paragraph`. Content-free structural tells: first *alphabetic* word lowercase (mid-sentence continuation), text ends with `,`/`;` or a coordinating conjunction, or a period-terminated sentence containing a lowercase non-stopword content word. Plus a **tunable recital-trigger** backstop — first word(s) ∈ a small universal set, default `{whereas, "now therefore"}` — the one sanctioned generic-keyword signal, overridable via `parse_to_opencontracts(parse_options={"recital_triggers": [...]})` → `to_opencontracts_export(recital_triggers=…)`. (This deliberately *does* match the canonical universal recital opener, unlike the older `_is_nonheading_furniture` stance above — a recital opener is universal contract boilerplate, not a corpus's words; the casing-count signal first tried here over-suppressed real statute/Title-case headings and was dropped.) The **root cause** is fixed in the engine too: `visual_ingestor.organize_and_indent_blocks` no longer font-promotes a block to `header` when `Line.is_prose_continuation` holds — a sanctioned engine edit (it also corrects the `json`/`html`/`all` renders; the S-1 baseline was re-floored accordingly). (The run-in box-split fix `Doc.split_runin_boxes` is a second such engine edit; the engine is not off-limits — see the convention note below.) `_is_centered_heading_line` also excludes `_is_prose_header` so the re-segmentation path cannot re-promote a demoted recital. Guard invariant (tests): real headings — ALL-CAPS, `ARTICLE N`, `§N. Words denoting…`, Title-Case-with-function-words, company-suffix titles (`… Inc.`), centered titles (`BYLAWS OF QUANTINUUM INC.`) — never demote. **Deferred** (separate, harder, context-dependent problems): the *table-row*-on-prose mislabel (justified recitals typed `table_row`) — every geometric guard tried corrupted real TOC/disposition tables — and *multi-column* reading order (the engine fuses 2-column pages). Design + outcome: `docs/superpowers/specs/2026-06-29-contract-pseudo-header-demotion-design.md`.
- **Document-level structural corrections** (`_structural_corrections`, computed once over all blocks for the cross-page / neighbor context the per-block `_resolve_label` lacks; returns a `{block_idx: "Paragraph"|"Title"}` override applied on top of `_resolve_label`): the **content-free, format-only** replacement for PR #9's reverted EDGARx2 *engine* rules (which had leaked into the json/all renders + the S-1 suite and content-matched SEC vocabulary). Four detectors, all keyed on geometry / repetition / raw `block_type` adjacency — never document words: (1) **repeated margin furniture** (`_furniture_demotions`) — a short (≤6-word) header repeated verbatim in a page margin across ≥`max(3, 4% of pages)` distinct pages is a running header/footer or per-page banner (subsumes running "TABLE OF CONTENTS" banners); a one-off real heading is never caught. (1b) **repeated-position furniture** (`_repeated_position_furniture`) — a header-ish block pinned at the **same bottom-right corner position across ≥2 pages** is a clerk/filing stamp or running corner mark; keys on repeated *geometry* (not text) so it catches the OCR-varying FortWorth "OFFICIAL RECORD / CITY SECRETARY" stamps the verbatim rule misses, while a single-page real bottom-right heading is structurally incapable of triggering it (legal-100 finding; the single-page geometric band was deliberately *not* widened — it would demote genuine bottom-right headings, which the hetero-100 suite caught). (2) **sparse-cover front-matter** (`_cover_overrides`) — on an early sparse page (≤12 blocks) with ≥2 centered pseudo-headers, the most prominent (largest font) becomes `Title` (an allowed parent, so a real cover title still anchors the tree) and the rest are demoted; left-aligned headings and dense first pages are untouched. (3) **embedded-in-table fragment** (`_embedded_fragment_demotions`) — a header interleaved within a Table Row run (immediate prev+next page blocks are Table Rows) and inside their column span is a mis-promoted table cell, demoted — **but only when it has a heading ancestor** (non-empty `level_chain`) so its children climb to that heading instead of orphaning into non-heading roots (the fw_vertigis quote-page guard). Keys on raw `block_type` (the `_LABEL_BY_BLOCK_TYPE` header set), not the resolved label, so it has no ordering dependency on `_resolve_label`; demotion takes precedence over `Title` promotion. Design: `docs/superpowers/specs/2026-06-29-edgarx2-hierarchy-structural-rehome-design.md`.
- **Tree consistency**: `parent_id` is the nearest level-chain ancestor whose *resolved* label is still a heading, so a demoted run-in header is spliced out and **no non-heading annotation ever bears children**.
- **Token recovery**: assignment is geometric (smallest containing rect), but tokens inside *no* rect (a table cell whose `box_style` undercovers its wrapped text) are recovered by `difflib`-aligning the page token stream to block word streams — *fills only*, never reassigns, so regression baselines are untouched.
- **Run-in token reconciliation** (`_reconcile_runin_overflow`): the exporter's **text-faithfulness guarantee**. `visual_ingestor` correctly splits a *run-in* heading ("6.6 Taxes. Each Party…", "SECTION 13. …" — heading + body on **one physical line**) into a header block + a paragraph block by *text*, but both are geometrically assigned the shared line-box, so the header's rect covers the body and it steals every token while the paragraph is left starved (sometimes zero) — a block's `tokensJsons` then spans a neighbour's text and the reconstructed text layer drifts from a blind flat extract. The reconciler re-partitions **by text in reading order**: a header block's trailing tokens its own `block_text` doesn't claim, that the *next* block's `block_text` begins with (and which that block is short of), are donated forward to it. Deliberately conservative so it is a strict no-op on clean/table/OCR/multi-column pages (baselines untouched): **only header-type donors**, donor must cover ≥75% of its own text, the donated run is a contiguous tail matching the next block's opening, and the next block must still be short of tokens. **Decoupled from labeling**: labels/hierarchy/`_structural_corrections` score on the pure-geometric assignment (`block_tokens_geo`), the repaired assignment (`block_tokens`) is what the *emitted* annotations carry — so faithfulness is fixed without shifting the tuned structural layer. `validate_export` now enforces a **fine-layer token partition** (no page token belongs to two structural annotations; the additive Semantic-Unit layer is excluded), and `_warn_unfaithful_blocks` logs (never raises) any residual overspan / zero-token block (the deferred multi-column signature-grid cases surface here). Verified on numbered-clause contracts (Eton, quantinuum/generate_bio charters): dozens of run-in clauses per page go from unfaithful to exact. **The *separate-line* run-in — heading on its own line, body below — is now root-fixed in the engine** (`Doc.split_runin_boxes`, called from the run-in split in `organize_and_indent_blocks`): the split used to hand the header and paragraph the *same* `BoxStyle` object (`~line 4202`, "Is this correct ???"); it now recomputes each block's span from its own visual lines, so no two blocks share a box and the `json`/`html`/`all` renders are correct too (verified S-1-clean — the engine's own cross-engine output only improves). The reconciler here then handles only the residual **same-line** run-in, where heading and body share *one* physical line: the engine's table-aware line-assembly collapses the per-word geometry a sub-box split would need (a 22-word run-in line reaches the split with far fewer word extents), so carving header/body boxes in the engine is unreliable — the exporter, which rebuilds clean per-word boxes from the XHTML PAWLS layer, is the robust place for it. A correct engine box changes a run-in header's `ntok`, so it no longer trips the absorbed-body demotion; a numbered **cross-reference** ("Section 201 of the … Act") that the buggy box used to demote *by accident* is now demoted structurally by `_is_cross_reference_header` (section word + enumerator immediately followed by "of"), which fixed the one vision-golden page (forbright p220) the correct boxes exposed. The correct boxes are a net structural win on human truth (legal-100 neutron_lime p221 head-ancestor 0→1.0, batch quantinuum p14 non-heading-roots 5→0); the only re-floored metric is the Docling oracle's head-ancestor on generate_bio (Docling under-labels the now-correctly-promoted numbered clauses — an oracle artifact, not a regression).
- **Block re-segmentation** (`_segment_block`): a non-header block that *begins* with centered heading line(s) (narrow + symmetric side margins, by token geometry) followed by left body is split into a `Section Header` + body annotation (heading parents body) — recovers a heading `visual_ingestor` fused into its following paragraph (centered title+preamble, "RECITALS"+WHEREAS). The granular per-visual-line truth is intact in the XHTML; only the block grouping was lossy, so the exporter re-splits without engine edits. Conservative: leading heading only, non-header blocks only (never referenced as parents), never makes an over-long header.
- **List-item → lead-in parenting** (`_list_intro_parents`): a contiguous run of `list_item` blocks whose immediately-preceding block is a colon-terminated `Paragraph` ("… agree as follows:") is parented to that paragraph (instead of the nearest header) — so enumerated clauses nest under the sentence that introduces them. This is the one sanctioned non-header parent; `diagnostics.childbearing_non_headers` allows a Paragraph parenting only List Items.
- **`OC_PARENT_CHILD` relationships** (`_parent_child_relationships`): the `parent_id` tree is *also* emitted into `export["relationships"]` as explicit edges (one per parent: source=[parent], target=[direct children], `structural: true`) — the OpenContracts convention honored by its subtree-group walker. `validate_export` checks relationship referential integrity. `oc_visualize.project_page` draws these as parent→child arrows (and an "↑parent [id] pN" cue for cross-page edges).

### Semantic-Unit grouping layer (`semantic_units=True`, opt-in, PDF only)
`ingestor/semantic_units.py::append_semantic_units(export)` is an **additive** post-processor that groups the fine structural annotations into coarser, **nested `"Semantic Unit"`** clause groupings (each an indemnity/MFN/subordination clause = a heading or numbered-clause root + its descendants). It reads only the finalized export and **appends** `"Semantic Unit"` annotations (ids `su-*`) + `OC_SEMANTIC_UNIT` (unit→member fine ids) and `OC_PARENT_CHILD` (unit→child-unit) relationships — it **never** mutates the fine layer, its `parent_id` tree, or the four locked baselines. **Units are flat: `parent_id` is always `null`** — the nested clause hierarchy is carried **entirely by the `OC_PARENT_CHILD` relationship edges** among `su-*` units (never the `parent_id` FK), membership by `OC_SEMANTIC_UNIT`, and unit identity by the `"Semantic Unit"` label tag. So a consumer reconstructs the whole clause tree from relationships + tags alone (roots = units that are no `OC_PARENT_CHILD` target). Gated by the `semantic_units` flag threaded through `parse_to_opencontracts(parse_options={"semantic_units": True})` / `to_opencontracts_export(..., semantic_units=True)` / the daemon's `semantic_units=true`; **default off** so the existing renders/suites are byte-identical. Deterministic B′ core: **coarsen** the engine heading tree, **guard** (demote folio/prose false-heading roots; a bare-ordinal `"2."` is folio furniture, not a clause), **numbering split** (break enumerated clause runs `1.`/`(a)`/`(i)` into nested per-clause sub-units by enumerator depth). Unit `annotation_json` bounds are the union of a unit's **direct members** only (a parent clause's box is its lead-in; each leaf clause's box is its own text) so nesting isn't mis-scored as merges and units align to a per-clause golden. This is the **foundational representation** a clause classifier/RAG-chunker consumes; clause *typing* is a deferred layer. Design + status: `docs/superpowers/specs/2026-07-01-semantic-unit-grouping-design.md`; Phase-1 plan: `docs/superpowers/plans/2026-07-01-semantic-unit-grouping-phase1.md`. The ONNX prose-boundary embedder (Phase 3) and CUAD clause-recovery eval (Phase 4) are deferred.

### Image-token layer (`include_images=True`, opt-in, PDF only)
`ingestor/image_tokens.py::extract_page_images(pdf_bytes, page_dims)` extracts embedded raster figures via pypdfium2 and the exporter appends them as `is_image: true` PAWLS tokens (base64 PNG, sha-256 `content_hash`, `image_type: "embedded"`; full-page scan backgrounds ≥85% of page area and <4pt slivers are filtered) **after** each page's text tokens, so text-token indices never shift. An image ≥90% inside an existing annotation's bounds joins it (`content_modalities: ["IMAGE","TEXT"]` — modality lists are sorted); otherwise a standalone `Image` annotation (`["IMAGE"]`) is emitted, parented like a text block at its position (preceding heading, or that block's parent). `validate_export` enforces the image-token field + modality invariants unconditionally; Semantic-Unit annotations take the sorted union of member modalities. Default **off** (flag-off byte-identical; the locked baselines are untouched — only `oc_export_baseline.json` covers it, via the `with_images.pdf` fixture + `oc_compat.FIXTURE_PARSE_OPTIONS`; regenerate the fixture with `scripts/build_image_fixture.py`, dev-group reportlab). Threaded like `semantic_units`: `parse_to_opencontracts(parse_options={"include_images": True})` / daemon `include_images=true`. Design: `docs/superpowers/specs/2026-07-03-oc-image-tokens-design.md` (resolves issue #1; vector graphics, tile merging, and `image_path` storage are recorded non-goals).

Diagnostics + visualization: `ingestor/oc_visualize.py` (`project_page`/`project_document` overlay annotation boxes on the PDF raster grouped by structural subtree; `render_tree_outline`/`diagnostics`/`page_summary` dump the `parent_id` tree + structural-health metrics). CLIs: `scripts/oc_visualize.py`, `scripts/oc_sample_audit.py`. Format reference: `docs/opencontracts_export_format.md`; design: `docs/superpowers/specs/2026-06-27-opencontracts-export-design.md`; layout study: `docs/docsling_layout_oracle.md`; heuristic audit: `docs/oc_export_heuristic_audit.md`; audit runbook (repeatable structural-quality audit + regression lock-in): `docs/runbooks/structural-quality-audit.md`.

### PDF pipeline (the core)
`pdf_ingestor.py::parse_pdf()` → `pdf_file_parser.parse_to_html()` (the **pure-Python** `PdfPlumberFileParser`) → `parse_blocks()`:
1. `file_parser/pdf_plumber_parser.py` opens the PDF with pdfplumber, groups words into visual-line segments, and emits **Tika-format XHTML**. Scanned/sparse pages (<4 text lines) are routed to `file_parser/ocr_parser.py` (rapidocr) which emits the *same* word-box format. Two gutter-aware routings fix multi-column welds *before* the engine sees the stream (the engine welds same-top stream-adjacent `<p>` segments into one block): (a) the **prose XY-cut** (`_detect_column_gutters` + `_group_words_into_lines_columns`, PR #17) splits genuine 2-column prose column-major; (b) the **grid-cell splitter** (`_detect_grid_regions` + `_apply_grid_regions`; tests in `tests/test_pdf_plumber_grid.py`) does the same for the two-column *form/signature/approval grids* the prose gates exclude — without it, left and right cells of the same row fuse into single blocks ("By:  By:", "Dana Burghdoff  Gary L. Wert"), which capped the semantic-unit clause benchmark on grid-heavy contracts and is unfixable downstream. A grid region is a banded run of ≥5 rows sharing exactly ONE trimmed interior gutter that survives width/balance gates *per zone* (kills 3-col tables, TOCs, label|value tables, in-column slivers), has ≥3 rows entirely on each side, and passes the **form gate**: colon-terminated label cells (`By:`, `Name:`, `VENDOR:` — punctuation *format*, never word content) on both sides, on ≥28% of rows, with median ≤5 words/cell — thresholds measured to separate the legal grid targets (form fraction 0.32–0.90) from the enterprise 2-column regions the hetero-100 suite guards (0.07–0.23); an early looser gate mis-fired on 49/100 hetero pages. Off-page word coordinates (rotated-sidebar bleed) disable the detector for that page. Applies on the text path (only when prose gutters found nothing) and the OCR path. Crossing rows flush full-width; regions never span a major vertical gap (pitch-aware banding).
2. `visual_ingestor/visual_ingestor.py::Doc(pages, ...)` is the rule-based engine (~6300 lines): groups visual lines into typed **blocks** (`header`, `para`, `list_item`, `table_row`), detects tables (using `<svg>` rule lines), strips repeating headers/footers, fixes reading order (`order_fixer.py`).
3. Header/indent levels via the engine's built-in `indent_parser.py` (the sole pathway — the optional `NewIndentParser` re-leveling pass and its `use_new_indent_parser` flag were removed 2026-07-02 after losing a golden A/B on hierarchy quality; levels are not scored by the S-1 suite, so no baseline moved).
4. `compress_blocks()` finalizes; `ingestor_utils/utils.blocks_to_sents()` derives sentences.

### The XHTML contract (most important thing to preserve)
`visual_ingestor/style_utils.py::parse_tika_style()` decodes each `<p>` element. The front-end MUST emit, per page:
```
<div class="page" style="height:Hpx; width:Wpx; position: relative;">
  <svg ...><line x1 y1 x2 y2/> ...</svg>      <!-- rule lines, for table detection -->
  <p style="height:..;font-size:Npx;font-family:F;font-style:..;font-weight:..;
            top:Ypx;position:absolute;text-indent:Xpx;
            word-start-positions:[(x,y), ...];        <!-- one (x,y) per whitespace token -->
            word-end-positions:[(x,y), ...];
            word-fonts:[(family,weight,style,size,size,space_width), ...]">text</p>
```
One `<p>` per visual-line segment; `text.split()` length must equal the number of word tuples; coordinates are absolute top-left-origin PDF points; `space_width = size/4`. To regenerate a gold reference from the old engine you'd need the (now removed) Tika jar — instead, the fixture tests + `parse_blocks` on the produced XHTML are the regression guard.

### Key shared pieces
- **`ingestor/table_engine.py`** — warp's own **table-cell engine** (pure MIT stack: pdfplumber ruled grids + envelope edges + whitespace-channel region grid inference, stacked-table splitting, ruled-run stitching, external-header attach). It is the **default** ParseBench table provider (`WARP_TABLE_PROVIDER=native`, incl. faithful mode) and lifted ParseBench Tables 36.5 → 57.4 (Overall 35.7 → 39.0). Render-boundary only — it never touches `visual_ingestor` or the S-1/OC exports. Unit tests: `tests/test_table_engine.py` (pure functions over word dicts). Design + eval loop: `docs/superpowers/specs/2026-07-02-native-table-engine-design.md`.
- **`ingestor/line_parser.py`** — the `Line` class is the central sentence/line classifier for the parsing pipeline. Per `CONTRIBUTING.md`, changes here **must** come with tests in `tests/test_line_parser.py`.
- **`ingestor_utils/compat.py`** — vendored helpers (`ensure_bool`, `generate_version`, `extract_file_properties`) that replaced the heavy external `nlm_utils` dependency. The package no longer imports `nlm_utils` anywhere.
- A **block** is a dict; common keys: `block_type`, `block_text`, `visual_lines`, `box_style`, `page_idx`, `block_idx`.

## Conventions & gotchas

- **Formatting is enforced**: black + isort (`profile=black`, line length 88). Run `make format` before committing or CI lint fails.
- **Preserve the XHTML contract in the *front-end*** (extraction/OCR): the pure-Python parser must emit the exact word-box XHTML the legacy Tika produced so `visual_ingestor` sees byte-identical input. That invariant is about the front-end↔engine boundary — it is **not** a blanket ban on editing the engine. **The layout engine is fair game, and a *systematic* engine flaw — a defect that recurs across a structural class rather than a one-off — should be fixed *in the engine*, at its root, not papered over in a downstream consumer.** The bar for an engine change is **no systematic degradation**: validate against the S-1 cross-engine regression (the engine's own json/block output — improvements pass, regressions fail) and the OC baselines, and **re-floor/regenerate a baseline when a correct fix legitimately shifts it** (that is the expected, sanctioned response, not something to avoid). Reach for a downstream/exporter-scoped correction (see the OpenContracts section) only when the fix genuinely belongs to that consumer, or when an engine change's blast radius isn't yet worth paying — and when you do, say so explicitly rather than defaulting to it because the engine "feels" off-limits.
- **OCR is optional**: `ocr_parser.ocr_available()` gates it; if `rapidocr-onnxruntime` isn't installed, scanned pages degrade to little/no text instead of crashing. `WARP_DISABLE_OCR=1` hard-disables OCR (env var, so it also reaches front-end worker processes). Default OCR render is 200 DPI (`ocr_parser.OCR_DPI`). rapidocr sometimes glues words on dense scans — a known limitation.
- **Parallelism & perf knobs** (all output-identical; see `benchmarks/PERF_OPTIMIZATION.md`): the front-end stripes pages of docs ≥8 pages across a persistent spawn process pool (`WARP_FE_WORKERS`, ≤1 = serial; `WARP_FE_PARALLEL_MIN_PAGES`); the engine shares pure `Line`/`Word` analyses via bounded LRU caches (`line_parser.bare_line`/`make_word`; `WARP_LINE_CACHE=0` bypasses). Engine callsites must use `line_parser.bare_line(text)` for bare single-string construction. Note monkeypatches don't reach spawned workers — use the env vars in benchmarks/tests (the timing harness does). The default runtime is **Python 3.14** (`python:3.14-bookworm` images; opt in locally with an untracked `.python-version`; 1.5× single-thread vs 3.12, hash-identical output); supported range stays 3.10–3.14 and the experimental JIT stays off.
- **Tuned magic constants** live at the top of `visual_ingestor.py` (e.g. `header_margin`, `line_height_threshold`, `table_col_threshold`) — several marked "don't touch this!". Changing them shifts parsing behavior globally.
- **Debug tracing**: `visual_ingestor.py`/`table_parser.py` have module-level `*_DEBUG` flags and a noisy `PROGRESS_DEBUG=True` (prints per-page progress) — flip rather than add prints.
- **`num_pages` is `len(pages) - 1`** in `parse_blocks` (an off-by-one inherited from the original; preserved for compatibility). Don't assert on it for single-page docs.
- **PDF-only**: the daemon and `ingest_document` accept `application/pdf` and raise `UnsupportedMediaType` otherwise. The earlier HTML/text/XML/Markdown ingestors were removed; don't reintroduce non-PDF parsing paths into the dispatcher.
